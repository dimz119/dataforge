# Phase 5 — Streaming Runtime (Kafka Backbone + REST Consumption)

**Deliverable:** D18 (phase doc)

Phase 5 ships the core product loop: a stream a user can start and stop, generated by leased runner processes reconciling desired state (ADR-0006), published through internal Kafka (ADR-0005), persisted into the time-partitioned REST buffer behind the `DeliveryChannel` interface, and pulled with a replayable cursor over an API key. The pipeline assumes its **final shape** here — behavior → ledger → (pass-through chaos slot) → internal Kafka → sinks — so chaos (Phase 9) and new channels (Phase 12) plug into existing seams instead of reshaping anything.

## Goal

> The core product loop: a continuously running, controllable stream through the final pipeline shape.

## Scope

- **Stream resource:** model + APIs — `POST /api/v1/streams` (T1: copies the instance pin, fixes the seed — client-supplied or generated, immutable thereafter per INV-STR-5), `POST /api/v1/streams/{id}/start` and `/stop` (idempotent, INV-STR-3), `GET` detail with lifecycle state + `status_reason`. `target_tps` is set at creation and applied at start; *live* TPS mutation is Phase 6.
- **Runner data plane (replaces the Phase-1 stub):** Redis lease acquisition per (stream, shard 0) with TTL 15 s / heartbeat 5 s and fencing tokens (INV-STR-2); desired-state poll + reconcile every tick; pool seeding on first start, checkpoint restore on restart/failover; periodic checkpoints every 30 s; `starting → running → stopping → stopped` convergence per domain-model §4.3, including T4 (`failed` when no lease within 60 s).
- **Kafka publication:** post-ledger envelopes produced to `df.delivery.events.v1` keyed by `partition_key` (workspace-prefixed, S-5); the chaos stage exists as an identity pass-through slot in the pipeline so Phase 9 inserts a transform, not a topology change.
- **`DeliveryChannel` interface + buffer-writer sink:** the frozen sink contract (`deliver(batch)`, ack/cursor semantics, backpressure signal); buffer-writer consumer group `df.sink.rest-buffer.v1` — batches ≤ 500 events / 250 ms linger (BW-2), `strip_internal()` at ingest (SB-2), transactional COPY into the time-partitioned `event_buffer`, offsets committed after the insert (at-least-once, INV-DEL-3); 24 h retention partitions with the drop job.
- **REST cursor pull:** `GET /api/v1/streams/{id}/events?from=earliest|cursor=…&limit=…` → `{data, next_cursor}`; opaque replay-stable cursors; `410` problem type `cursor-expired` with `earliest_cursor` in the body (INV-DEL-4); API-key auth requiring `events:read`; basic per-key rate limiting.
- **Celery control plane:** lifecycle command handling (desired-state writes + audit entries), lease-expiry watchdog feeding T4/T11, partition-maintenance beat jobs.
- **Permanent gates extended:** TEN data-plane probes (foreign key on a stream → 404; revoked-scope variant → 403) and the SB-3 strip scan over delivered REST output join CI forever.
- **Connection guide G1** (REST → user's local Kafka bridge — exercise E8's MVP form) published with the documented poll loop and `410` handling.

## Non-goals

| Deferred | Lands in |
|---|---|
| Pause/resume, live TPS changes, WS delivery, stream stats | Phase 6 |
| Chaos transforms (the stage slot is pass-through) | Phase 9 |
| Multi-shard streams (`shard_id = 0` everywhere; shard-count field exists, pinned to 1) | Phase 11 |
| Quota metering / idle auto-pause (command-time caps only) | Phase 11 |
| Hosted external Kafka, webhooks, S3 export (the sink interface is the seam) | Phase 12+ |
| Backfill-mode streams (batch backfill exists from Phase 4; stream `clock_mode=backfill` realism work) | Phase 8 |

## Demo script

1. `docker compose -f infra/compose/compose.yaml up -d --wait --scale runner=2` — two runner replicas for the failover demo.
2. Auth + instance: `infra/scripts/demo-phase02.sh` for `$ACCESS`/`$WS`/`$KEY`; create `$INST` as in phase-04 demo step 2.
3. Create stream: `STREAM=$(curl -s -X POST localhost:8000/api/v1/streams -H "Authorization: Bearer $ACCESS" -d '{"workspace_id":"'$WS'","scenario_instance_id":"'$INST'","seed":4242,"target_tps":10}' | jq -r .stream_id)`.
4. Start: `curl -s -X POST localhost:8000/api/v1/streams/$STREAM/start -H "Authorization: Bearer $ACCESS"`; poll `GET /api/v1/streams/$STREAM` until `status: "running"`.
5. Pull events with the API key: `curl -s -H "X-API-Key: $KEY" "localhost:8000/api/v1/streams/$STREAM/events?from=earliest&limit=200" | jq -r '.data[].event_type' | sort | uniq -c` — session, order, payment events flowing.
6. Referential check: pick an `order_placed` payload's `user_id` and confirm a prior event/snapshot position references that user — Orders reference previously emitted Users/Products.
7. Cursor replay: capture `next_cursor` `C1`; fetch the same `C1` page twice; `diff` of the two response bodies → identical (INV-DEL-3).
8. Idempotency: `POST …/start` again → 200 returning current state, no second runner claim.
9. Kill-test: identify the lease-holding runner (`docker compose ps`, runner logs), `docker compose kill <holder>`; watch events resume — gap between last pre-kill and first post-failover `emitted_at` **< 30 s**; canonical `sequence_no` continues gaplessly.
10. Stop: `T0=$(date +%s); curl -s -X POST …/streams/$STREAM/stop -H "Authorization: Bearer $ACCESS"`; poll the events frontier — last delivered `emitted_at` within **5 s** of `T0`; status reaches `stopped`.
11. Cross-tenant: workspace-B key on `GET …/streams/$STREAM/events` → `404`.
12. Strip check: `curl -s … "?from=earliest&limit=500" | grep -c '_df'` → `0`.
