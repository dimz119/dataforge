# ADR-0014 — API conventions: `/api/v1`, cursor pagination, RFC 9457 errors, OpenAPI as CI artifact

**Deliverable:** D17

Every DataForge HTTP surface follows one set of conventions: URL-versioned routes under `/api/v1`, opaque-cursor pagination on every list and event endpoint, RFC 9457 problem-details for every error, per-key rate limits, and an OpenAPI schema generated by drf-spectacular as a CI artifact from which the frontend's TypeScript client is generated — so FE/BE contract drift fails the build. This warranted an ADR because API conventions are unfixable after external consumers exist: learners script against these endpoints from Phase 5, and an inconsistency (two pagination idioms, ad-hoc error shapes) becomes permanent documentation debt and broken-client risk.

- **Status:** Accepted
- **Date:** 2026-06-10
- **Decides for:** every REST endpoint in every phase; the error contract of every channel's HTTP surface; the FE/BE contract pipeline (ADR-0001, ADR-0016); public API docs

## Context

The forces:

- **The API is a teaching surface.** The software-engineer persona learns "event envelopes, idempotency, ordering guarantees, and at-least-once semantics by experiencing them" through "a well-documented event API" (PRD §2.5). Conventions must be standard-shaped, curl-obvious, and identical everywhere — a learner who masters one endpoint has mastered them all.
- **Contract stability is an NFR** (PRD §9): `/api/v1` versioning, cursor pagination, RFC 9457 errors, and OpenAPI-as-CI-artifact are the named mechanism. The envelope evolution policy (event-model §8, EV-5) explicitly couples envelope major versions to API major versions — `/api/v1` must therefore be additively evolvable for the platform's lifetime.
- **The event-pull endpoint forces the pagination decision.** The REST channel is a cursor over a continuously appended, partition-dropped buffer (ADR-0013). Offset pagination over that store is incoherent (rows shift under the reader, deep offsets scan partitions); whatever idiom events use should be the only idiom, or client code forks.
- **One repo, two apps, one contract** (ADR-0001): the OpenAPI schema and the generated client are how `backend/` and `frontend/` evolve in lockstep; drift must be a build failure, not a runtime surprise.
- **Free-tier compute reality:** per-key and per-IP rate limits are a security control (TM-5, [../06-quality/security-architecture.md](../06-quality/security-architecture.md) §7), and their error semantics belong to the API conventions, not to each endpoint's whim.

## Decision

1. **URL versioning.** All REST routes live under `/api/v1/`; WS routes under `/ws/` carry a versioned subprotocol instead (ADR-0013). Within `v1`, evolution is additive only: new endpoints, new optional request fields, new response fields. Nothing is removed, renamed, retyped, or re-semanticized. `/api/v2` is reserved for a breaking change and is deliberately expensive (it would entail a new envelope major and WS subprotocol per EV-5) — the cost is stated so the change is effectively never made.
2. **Cursor pagination everywhere.** Every collection response has the shape `{"data": [...], "next_cursor": "<token>" | null}`. Cursors are opaque, URL-safe tokens passed back via the `cursor` query parameter; page size via `limit` (server-clamped per endpoint, defaults owned by [../05-interfaces/api-specification.md](../05-interfaces/api-specification.md)). No `offset`, no page numbers, on any endpoint — control-plane lists (workspaces, keys, streams, audit entries, injections) use the same idiom as the event pull, so one client pagination helper serves the whole API. Event-buffer cursors additionally carry the replay semantics of ADR-0013 (INV-DEL-3/4).
3. **RFC 9457 problem details for every error.** Content type `application/problem+json`; fields `type` (URI under `https://docs.dataforge.dev/problems/{slug}`), `title`, `status`, `detail`, `instance` (the request path), plus problem-specific extensions (e.g. `errors[]` with field paths on validation failures, `retry_after` seconds on rate limits, the expiry horizon on `cursor-expired`). The problem-type catalog — including `validation-error` (400), `authentication-failed` (401), `permission-denied` (403), `not-found` (404), `conflict` (409), `cursor-expired` (410), `quota-exceeded` (422), `rate-limited` (429) — is owned and enumerated by [../05-interfaces/api-specification.md](../05-interfaces/api-specification.md). No endpoint ever returns an ad-hoc error body; `DEBUG = False` deployments never leak stack traces (SEC-TLS-4).
4. **Per-key rate limits.** Redis token buckets keyed by API key (data plane) or user/IP (console/auth surfaces), enforced in middleware before view code, returning `429` with `Retry-After` and problem type `rate-limited`. The limit catalog (RL-1…RL-9) is owned by the security architecture §7; this ADR fixes the *semantics*: limits are per-principal, responses are uniform, and trips are audited in aggregate.
5. **OpenAPI as a CI artifact.** drf-spectacular generates `openapi.json` in CI on every commit (ADR-0001). The frontend generates its typed client from that artifact (`openapi-typescript` + `openapi-fetch`, [../02-architecture/frontend-architecture.md](../02-architecture/frontend-architecture.md) §5); the drift gate regenerates `schema.gen.ts` and fails the build on any diff. Serializers, problem types, and pagination shapes are annotated so the schema is complete — an endpoint that cannot be expressed in the schema does not ship.

The conventions at a glance:

| Convention | Rule | Enforced by |
|---|---|---|
| Versioning | `/api/v1` prefix; additive-only within v1; `v2` ≈ never (EV-5 coupling) | API spec review; envelope policy (event-model §8) |
| Pagination | `{"data": [...], "next_cursor": ...}`, opaque `cursor` + clamped `limit`, on every collection | Contract tests; generated client types |
| Errors | RFC 9457 `application/problem+json`, catalogued `type` URIs, no ad-hoc bodies | Single DRF exception handler; contract tests |
| Rate limits | Redis token buckets per principal; `429` + `Retry-After`, problem type `rate-limited` | Middleware before view code (security architecture §7) |
| Contract | drf-spectacular `openapi.json` CI artifact → generated TS client; diff fails build | CI drift gate from Phase 1 |

## Alternatives considered

- **Header or media-type versioning** (`Accept: application/vnd.dataforge.v1+json`). Rejected: invisible in `curl` examples, browser dev tools, and copy-pasted student snippets; harder to route/cache; generated-client tooling handles URL prefixes trivially. For a teaching API, the version belongs where learners can see it.
- **Offset/page-number pagination** (DRF's default `PageNumberPagination`/`LimitOffsetPagination`) — at least for control-plane lists. Rejected: unstable under concurrent inserts (every list in DataForge is append-heavy — events, audit, injections), O(partition-scan) at deep offsets on the time-partitioned buffer, and it would split the API into two idioms with two client code paths. Cursor pagination costs "jump to page N" (accepted: the console uses infinite scroll/next-prev, frontend-architecture §4) and buys stability everywhere.
- **Ad-hoc JSON error envelopes** (`{"error": "...", "code": 42}`). Rejected: every shape decision (field names, where details go, how field errors nest) gets re-litigated per endpoint; RFC 9457 is the published standard, has client library support, gives stable machine-readable `type` URIs for the teaching-relevant errors (`cursor-expired` is a documented lesson, PRD §7), and maps onto DRF exception handling with one handler.
- **Hand-written TypeScript client / hand-maintained OpenAPI document.** Rejected: hand-maintained contracts drift silently — the precise failure mode ADR-0001 exists to prevent. Generation makes drift a compile error in the same PR that caused it.
- **GraphQL.** Rejected: the stack mandates DRF REST; event consumption is stream-shaped (cursor over append order), not query-shaped; per-key rate limiting and problem semantics are HTTP-native here; and a GraphQL layer would be a second contract surface to keep drift-free for zero persona benefit.

## Consequences

### Positive

- One pagination idiom, one error shape, one auth header convention across ~40 endpoints: client code, docs, and tests share helpers; learners transfer knowledge across the whole API.
- Contract drift is structurally impossible to merge: the OpenAPI artifact + generated-client diff gate runs from Phase 1 ([../07-plan/phases/phase-01-foundations.md](../07-plan/phases/phase-01-foundations.md)).
- Problem-type URIs give the docs site stable anchors — every API error a student hits links to an explanation, including the deliberate teaching errors (`410 cursor-expired`, `429 rate-limited`).

### Negative

- Cursor pagination forfeits random page access and server-computed total counts on large collections; list UIs are designed around it (accepted in frontend-architecture §4).
- drf-spectacular completeness requires annotation discipline — un-annotated serializers produce wrong schemas silently; mitigated by the drift gate plus a CI schema-validity check (no `unknown` types allowed in the artifact).
- URL versioning means a hypothetical `v2` duplicates route tables; accepted, because EV-5 makes `v2` a never-event by policy.
